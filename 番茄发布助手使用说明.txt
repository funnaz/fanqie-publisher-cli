番茄小说半自动发布助手 CLI 使用手册

一、工具用途

这个工具用于把任意本地 txt 小说章节半自动填入番茄作家后台。

它不会保存账号密码，也不调用非公开接口。浏览器打开后，需要你自己登录番茄作家后台，并进入目标作品的章节页面。

当前已测试跑通的能力：
1. 自动扫描浏览器标签页，找到真正的章节编辑页。
2. 自动填写左侧章节号。
3. 自动填写右侧章节名。
4. 自动填写正文到编辑器。
5. 自动删除正文第一行的“第x章 章节名”。
6. 自动保存草稿。
7. 保存后切换到新的章节编辑页，避免覆盖上一章。
8. 从草稿箱/章节管理页按章节范围发布已保存草稿。
9. 发布草稿时会从草稿箱列表点击编辑图标进入草稿编辑页。
10. 每发布一章后会自动切回草稿箱，继续下一章。
11. 支持任意小说章节目录。
12. 支持 JSON 配置文件。
13. 支持断点续传、失败截图、页面诊断。


二、章节目录要求

推荐结构：
某本书
  chapters
    001-第1章 开篇标题.txt
    002-第2章 第二章标题.txt

也支持每 20 章一个子文件夹：
某本书
  chapters
    001-020
      001-第1章 开篇标题.txt
      002-第2章 第二章标题.txt

文件名必须以章节号开头：
001-
002-
003-

正文第一行推荐写：
第1章 开篇标题

后面空一行，再写正文。

脚本会自动拆成：
章节号：1
章节名：开篇标题
正文：删除第一行，只上传正文内容


三、安装依赖

在项目目录执行：
npm install


三点五、Web 控制台

启动：
npm run web

打开：
http://localhost:3899

Web 控制台可以：
1. 填写章节目录、书名、起止章节和运行模式。
2. 启动“上传草稿”“发布草稿箱”“本地检查”等任务。
3. 用“继续 / 回车”按钮响应命令行等待。
4. 用“停止”按钮终止当前任务。
5. 实时查看运行日志。
6. 查看当前章节目录草稿上传到哪一章、发布到哪一章。
7. 创建定时发布任务，每 30/60/120/240 分钟自动发布下一批草稿章节。

注意：
浏览器登录和番茄后台页面仍然由 Playwright 打开，你需要在打开的浏览器里手动登录。

定时发布说明：
1. 先填写章节目录、书名、截止章节。
2. 选择 30/60/120/240 分钟间隔。
3. 设置每次发布章节数。
4. 点击“创建定时发布”。
5. 到时间后，系统会读取当前“已发布到第几章”，自动发布下一批。
6. 如果当前已有任务运行，本次定时发布会跳过，等待下一次。
7. 定时任务保存在本地 .fanqie-schedules.json，不会上传到 GitHub。


四、两种使用方式

方式 A：每次命令指定章节目录

本地检查：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --dry-run --start 1 --end 10

保存草稿：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --draft --start 1 --end 3 --confirm-each --reset

方式 B：使用配置文件

复制：
fanqie.config.example.json

改名为：
fanqie.config.json

示例配置：
{
  "book": "示例书名",
  "chapters": "D:/novels/示例书名/chapters",
  "url": "https://fanqienovel.com/writer/zone",
  "newUrl": "",
  "minChars": 950,
  "delay": 1200,
  "confirmEvery": 5,
  "confirmEach": false,
  "strictQuality": true
}

使用配置检查：
npm run fanqie -- --config ".\fanqie.config.json" --dry-run --start 1 --end 10

使用配置保存草稿：
npm run fanqie -- --config ".\fanqie.config.json" --draft --start 1 --end 3 --confirm-each --reset


五、发布前检查

只检查本地章节，不打开浏览器：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --dry-run --start 1 --end 10

检查内容：
1. 是否缺章。
2. 是否重复章节号。
3. 标题是否为空。
4. 正文是否为空。
5. 正文字数是否低于最低字数。
6. 是否有明显重复段落。
7. 相邻章节相似度是否过高。

修改最低字数：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --dry-run --start 1 --end 10 --min-chars 1000


六、草稿上传流程

推荐先小范围测试：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --draft --start 1 --end 3 --confirm-each --reset

操作步骤：
1. 命令窗口打开后，浏览器会自动打开番茄作家后台。
2. 你手动登录。
3. 进入目标作品的章节管理页。
4. 推荐从章节管理页打开“新建章节”编辑页；如果忘记打开，脚本会尝试自动点击“新建章节”进入写入页。
5. 回到命令窗口按回车。
6. 脚本会自动填章节号、章节名、正文。
7. 如果带了 --confirm-each，每章填完会暂停，你检查后按回车保存草稿。
8. 保存后，脚本会关闭当前编辑页，并尝试打开新的章节编辑页。


七、避免覆盖上一章

保存草稿后，当前页面通常会变成“编辑已有草稿”的页面。如果继续使用这个页面，会覆盖上一章。

脚本有两种处理方式。

方式 A：保留章节管理页

这是已测试成功的方式。

要求：
1. 浏览器里保留一个“章节管理页”标签。
2. 另一个标签是当前“新建章节”编辑页。
3. 保存一章后，脚本会尝试回章节管理页点击“新建章节”，再继续下一章。

如果第一次按回车时仍停留在主页或章节管理页，脚本会自动尝试点击“新建章节”进入写入页。

方式 B：指定真正的新建章节 URL

如果你能拿到番茄后台“新建章节”的固定 URL，用这个方式：
npm run fanqie -- --config ".\fanqie.config.json" --draft --start 1 --end 3 --confirm-each --reset --new-url "真正的新建章节URL"

注意：
不要把保存后的草稿编辑 URL 当作 --new-url，否则仍然会覆盖。


八、常用命令

检查前 10 章：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --dry-run --start 1 --end 10

保存前 3 章草稿，每章确认：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --draft --start 1 --end 3 --confirm-each --reset

保存 11-20 章草稿，每 5 章暂停：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --draft --start 11 --end 20 --confirm-every 5 --reset

继续 21-50 章草稿，每 10 章暂停：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --draft --start 21 --end 50 --confirm-every 10 --reset

正式发布模式：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --publish --start 1 --end 3 --confirm-each

正式发布前建议先保存草稿并人工检查，不建议直接大批量发布。

直接发布（先上传再发布）：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --upload-and-publish --start 1 --end 3 --confirm-each --reset

说明：
--upload-and-publish 会先执行上传流程，填写章节号、章节名和正文，然后点击发布，并处理后续弹窗。
--publish 只适合当前页面已经填好内容时使用，不会自动上传正文。

从草稿箱发布已保存草稿：
npm run fanqie -- --chapters "D:\novels\某本书\chapters" --publish-drafts --start 1 --end 3 --confirm-each --reset

草稿发布模式操作：
1. 浏览器打开后，进入番茄后台的章节管理页。
2. 脚本会优先选择“章节管理/草稿箱/新建章节”所在页面，不会使用已发布章节编辑页。
3. 脚本会先点击一次“草稿箱”，确认进入草稿箱页面。
4. 如果当前草稿箱页没有目标章节，会自动尝试翻页继续查找。
5. 回命令窗口按回车。
6. 脚本会按章节号和章节名寻找对应草稿行。
7. 找到后点击该行右侧的编辑图标，进入草稿编辑页。
8. 在草稿编辑页校验章节号和章节名，匹配后点击“下一步/发布”。
9. 如果弹出错别字提示，会等待 2 秒后精确点击“提交”。
10. 如果弹出内容检测方式，会等待 2 秒后精确点击“仅基础检测”。
11. 如果弹出发布设置，会等待 2 秒后选择“是否使用AI：是”，再等待 2 秒后精确点击“确认发布”。
12. 发布完成后如果页面回到章节管理页，下一章前会自动切回“草稿箱”。
13. 如果翻页后仍找不到章节或按钮，会截图并暂停，提示你手动处理。

已验证通过：
第 15-17 章草稿箱发布流程已测试成功。


九、断点续传

脚本会把成功处理过的章节记录到：
.fanqie-runs

断点记录会按书名、章节目录、章节范围、模式区分。

同样命令再次运行时，会跳过已完成章节。

从头重跑当前范围：
npm run fanqie -- --config ".\fanqie.config.json" --draft --start 1 --end 3 --confirm-each --reset

忽略断点但不删除记录：
npm run fanqie -- --config ".\fanqie.config.json" --draft --start 1 --end 3 --confirm-each --no-resume


十、页面诊断

如果脚本找不到输入框或编辑器，用诊断模式：
npm run fanqie -- --config ".\fanqie.config.json" --draft --start 1 --end 1 --inspect-page --reset

操作：
1. 浏览器打开后，进入番茄章节编辑页。
2. 回命令窗口按回车。
3. 脚本不会填写任何内容，只会导出诊断文件。

诊断文件位置：
.fanqie-runs

诊断会输出：
1. 脚本看到了几个 input。
2. 每个 input 的 class、placeholder、value、位置。
3. 看到了几个编辑器。
4. 当前页面截图和 HTML。

当前已确认的番茄编辑页结构：
章节号输入框：
input.serial-input.byte-input.byte-input-size-default

章节名输入框：
input.serial-input.serial-editor-input-hint-area.byte-input.byte-input-size-default

正文编辑器：
.ProseMirror[contenteditable="true"]


十一、参数说明

--chapters
章节目录。通常指向某本书的 chapters 文件夹。

--config
JSON 配置文件路径。

--book
书名，用于日志和断点标识。不传时从 chapters 路径推断。

--start
起始章节号。

--end
结束章节号。

--dry-run
只做本地检查，不打开浏览器，不上传。

--draft
保存草稿模式。

--publish
正式发布模式。

--upload-and-publish
先上传章节内容，再直接发布。

--publish-drafts
从草稿箱或章节管理页按章节范围发布已保存草稿。

--confirm-each
每章填完暂停确认。

--confirm-every 5
每 5 章暂停确认一次。

--reset
清空当前范围的断点记录，从头执行。

--no-resume
忽略断点记录，但不删除记录。

--min-chars 950
设置最低正文字数。

--delay 1200
每章提交后等待的毫秒数。

--url
打开的后台入口地址。

--new-url
真正的新建章节 URL。用于保存后打开下一章新建页，避免覆盖旧草稿。

--inspect-page
只导出页面诊断，不填写内容。

--no-strict-quality
忽略质量检查里的警告，但不能忽略错误。


十二、推荐使用策略

1. 每换一本小说，先设置 --chapters 或新建一个 fanqie.config.json。
2. 每次先 dry-run。
3. 首次只跑 1-3 章，并使用 --confirm-each。
4. 跑通后按 10 章或 20 章一批。
5. 大批量上传时使用 --confirm-every 5 或 --confirm-every 10。
6. 保留章节管理页标签，避免保存后找不到新建入口。
7. 发布前先全部保存为草稿，再在番茄后台人工检查。
8. 发布草稿时建议从草稿箱列表页开始，便于脚本按章节行定位。


十三、常见问题

问题：脚本显示 0 个 input、0 个编辑器。
原因：脚本没有控制到真正的编辑页，或者当前标签不是编辑页。
处理：运行 --inspect-page，确认浏览器里打开的是章节编辑页。

问题：保存后下一章覆盖了上一章。
原因：使用了保存后的草稿编辑 URL。
处理：保留章节管理页，让脚本点击“新建章节”；或提供真正的新建章节 URL 给 --new-url。

问题：章节标题出现在正文第一行。
处理：当前脚本会自动删除正文第一行“第x章 标题”。如果页面改版导致失败，运行 --inspect-page 重新定位编辑器。

问题：找不到保存按钮。
处理：脚本会保存失败截图和 HTML 到 .fanqie-runs。根据截图确认按钮文案是否变了。

问题：草稿箱里没有文字“发布”按钮。
原因：番茄草稿箱列表用的是编辑图标。
处理：当前脚本会点击草稿行右侧的编辑图标进入编辑页，再执行发布流程。

问题：发布一章后跳回章节管理页。
处理：当前脚本会在下一章开始前自动点击“草稿箱”。

问题：不想自动跳过已完成章节。
处理：加 --reset 或 --no-resume。
